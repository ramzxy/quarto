
================================ NETWORKING (Base Classes) ================================

SocketConnection (abstract)
    Base class for all network connections. Handles socket I/O, message
    reading/writing, and connection lifecycle. Subclasses implement
    handleMessage() and handleDisconnect() for protocol-specific logic.

SocketServer (abstract)
    Base class for servers. Handles ServerSocket binding and connection
    acceptance. Subclasses implement handleConnection() to create handlers.

================================ MODEL (Shared) ================================

Game
    The main game logic. Manages the board, pieces, players, and turns.
    Validates moves, checks for win conditions, and notifies listeners.
    Has a no-arg constructor for client-side use.

Board
    Represents the 4x4 game board. Stores placed pieces and checks for
    winning lines (4 pieces sharing a common trait).

Piece
    Represents one of the 16 Quarto pieces. Each piece has 4 boolean
    properties: tall/short, solid/hollow, dark/light, round/square.

Move
    Represents a player's action: placing the current piece on a board
    position and selecting the next piece for the opponent.

GameListener (interface)
    Callback interface for game events. Implemented by ClientHandler to
    receive move notifications and game-over events.

AbstractPlayer (abstract)
    Base class for all player types. Defines determineMove() which
    returns the player's chosen move given the current game state.

================================ SERVER ================================

Server (extends SocketServer)
    Entry point for the server. Accepts TCP connections on a port,
    creates a ClientHandler for each client via handleConnection().

GameManager
    Central coordinator. Handles username registration, matchmaking queue,
    and active game tracking. Pairs players and creates games.

ServerConnection (extends SocketConnection)
    Handles protocol parsing for server-side connections. Parses incoming
    messages and delegates to ClientHandler via receive*() methods.
    Provides send*() methods for outgoing protocol messages.

ClientHandler
    One per connected client. Contains business logic for handling
    protocol commands. Uses composition with ServerConnection.
    Implements GameListener to forward game events to the client.

ClientState (enum)
    Connection state: CONNECTED → HELLO_RECEIVED → LOGGED_IN → IN_QUEUE → IN_GAME

================================ CLIENT ================================

ClientApplication
    Entry point for the client. Creates the player, view, and GameClient,
    then starts the connection.

ClientConnection (extends SocketConnection)
    Handles protocol parsing for client-side connections. Parses incoming
    server messages and delegates to GameClient via receive*() methods.
    Provides send*() methods for outgoing protocol messages.

GameClient
    Main client orchestrator. Uses composition with ClientConnection.
    Manages the local game state. Supports multiple games on the same
    connection by resetting state when GAMEOVER is received.

HumanPlayer (extends AbstractPlayer)
    A player controlled by a human. When determineMove() is called,
    it prompts the user via ClientView and returns their input.

ComputerPlayer (extends AbstractPlayer)
    An AI-controlled player. When determineMove() is called,
    it delegates to a Strategy to compute the best move.

ClientView (interface) (in Client.views)
    Interface for displaying the game and getting user input.
    Includes methods for login, errors, game events, etc.

TUI (implements ClientView) (in Client.views)
    Text-based user interface. Displays the board in the console
    and reads user input from stdin.

================================ AI ================================

Strategy (interface)
    Interface for AI move computation. Takes the current game state
    and returns the best move according to the strategy.

ChokerJokerStrategy (implements Strategy)
    Advanced AI implementing a "Constraint-Solver" Hybrid Engine. Uses
    Strangler (heuristic) and God Engine (perfect play) phases, with
    bitboard representation, symmetries, and Zobrist hashing.

MinimaxStrategy (implements Strategy)
    AI using minimax algorithm with alpha-beta pruning.
    Evaluates positions and looks ahead multiple moves.

================================ DESIGN PATTERN ================================

Delegation Pattern:
  - ServerConnection parses protocol → calls ClientHandler.receive*()
  - ClientConnection parses protocol → calls GameClient.receive*()
  
This separates networking from business logic and allows:
  - Reusable connection classes
  - Easy testing of handlers
  - Same connection supports multiple games
